## 2026-03-22 14:00〜22:23

**実施内容**
- 制約ごとの `bump_activity` 特殊化を3制約で検討・実装
  - **AllDifferentExcept0**: 4ケースに特殊化（Hall pair 失敗、val=0 pool 枯渇、非0値重複の 70/30 分配、デフォルト）→ コミット済み
  - **NValue**: 3ケースに特殊化（def > n_max で unique contributor bump、poss < n_min で most-shared 値 bump、n 変数トリガー）→ コミット済み
  - **BoolClause**: 3手法を試行（全リテラル VSIDS 風、trigger 50%/残り50%、デフォルト+trigger 追加）→ いずれも悪化しリバート
- ベンチマークスクリプト2本を新規作成
  - `bench_nvalue.py` — nvalue 制約を含む問題の Sabori vs CP-SAT 比較
  - `bench_bool_clause.py` — bool_clause 制約を含む問題の比較（FlatZinc compile で検出）

**設計判断・理由**
- AllDifferentExcept0: 値重複時に未確定変数（conflicting value をドメインに持つもの）にも 30% を分配し、次の判定で優先されるようにした。Hall pair 失敗時はデフォルト（全確定変数）ではなく、pair に関与する変数のみに集中
- NValue: 失敗原因を `definite_count` vs `possible_count` vs `n_max`/`n_min` の関係で分類し、原因変数を特定
- BoolClause をリバートした理由: bool_clause は問題あたり数百〜数千の節が存在し、各節2-3リテラルと短い。節単位で変数に偏った bump を与えると大量の節からの累積で activity が暴れ、変数選択が悪化した（hitori で fail 568K→1030K に増加）
