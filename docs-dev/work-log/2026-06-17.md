## 2026-06-17 (前回までの続き〜11:19) LCG 補助変数昇格 (-A) の実装と多方向検証

### 実施内容

- **`-A` (promote learned-aux) を実装** (温存)。学習 nogood に現れる従属変数(`is_defined_var`)を
  決定 tier に昇格し、変数選択 tier-1 で decision var と activity 同列競合させる。
  - `VariableSelector`: `select_linear` を状態引き継ぎ式 `scan_candidates` に作り替え、
    tier-1 で decision ゾーン + `promoted_` を統合スコアリング。`update_promoted` で昇格集合を構築。
  - 昇格集合は `NoGoodManager::mark_nogood_vars` → `refresh_promoted` で **リスタート粒度で再計算**
    (`rebuild_var_ng_blooms` 直後)。GC で nogood が消えれば自然に降格。
  - `-L` と独立フラグ。partition 境界・backtrack 復元には未介入(健全性に触れない)。
  - 全 202 テスト pass、default mode 不変、決定論的、健全(最適値/最適解が -L と一致)。

- **見込み計測 (一時診断)**: 学習 nogood に現れる変数の 52〜81% が従属変数、
  conflict 由来 activity の 15〜41% が selector が後回しにする従属変数に座礁(hitori 40%)。
  → 仮説「bump はされるが decision でないと死蔵」を定量化。

- **早期効果**: 最適性証明までの fails で hitori −31% / steiner −16% (両方最適証明・同一最適値)。
  traveling-tppv は中立。**ただし cg2023(proof系)では -A が from-start の証明を破壊**
  (-L 6865 OPTIMAL → -L -A TIMEOUT)。

- **以前却下された2案を再検証** (ユーザ提起):
  1. **遅延起動 (proof-burst)**: env `SABORI_LEARN_DELAY=K` で実装し検証 → **前提は変わらず**。
     -A は proof を助けず、遅延起動を救わない(D2/D5 とも証明不成立)。
  2. **事前タイプ判別**: 静的特徴に加え **ランタイム信号**(explained比/prune率/act_def_share)でも
     非分離。「-A が害になる」型が異質(cg2023 explained=0.076 vs fillomino=0.927)で
     信号空間上で1クラスタにならない。価値の天井も低い(2026-06-14 regret 分析と整合)。

- **-L なしの -A 挙動 (aux 分岐単独) を検証** (env `SABORI_PROMOTE_ALL` / `SABORI_PROMOTE_BOOL`):
  - promote-all(学習なし全 defined 昇格): fails −6〜17% だが **壁時計は損**
    (steiner −17% fails / +25% wall)。tier-1 スキャン爆発が削減を相殺。
  - promote-bool(bool defined のみ): steiner は promote-all と同等だが
    **hitori は fails 2.4倍・壁時計2倍弱の大退行**。bool/int を一緒に候補化する balance が重要で、
    片方に絞ると分岐順が歪む。
  → aux 昇格はどの絞り方でも問題依存・予測不能。

- **クリーンアップ**: 実験スカフォールド全撤去 (`SABORI_LEARN_DELAY` / `SABORI_PROMOTE_ALL` /
  `SABORI_PROMOTE_BOOL` / `-v aux-diag` / `diagnose_aux` / Model `is_bool` 配線)。
  **`-A` 本体 (nogood-based promote) のみ温存**。撤去後 env は inert、202 テスト pass を確認。

### 設計判断・理由

- 昇格は **partition 物理移動でなくサイド集合 `promoted_`** で実現。`decision_var_end_` 等は
  探索中不変前提でコード全体が書かれており、動的移動は backtrack 復元と絡んで高リスク。
  サイド集合なら trail/backtrack と decouple し健全性リスク最小。
- 昇格集合の再計算は **per-clause refcount でなくリスタート粒度バッチ**。`rebuild_var_ng_blooms`
  が既に全 nogood×全リテラルを走査しており追加コストゼロ。GC もリスタート時なので粒度が合う。
- **`-A` を温存する理由**: 学習が既に効く問題(hitori/steiner)では明確な search 削減があり、
  opt-in フラグ(default off)で default 挙動に影響しない。ただし proof 系では諸刃。

## 2026-06-17 14:00〜15:52 片側定数 reif bool の決定変数昇格 (-R) 実装とバンドチューニング

### 実施内容

- **`-R` (promote const-side reif bools) を実装**（温存、デフォルト off）。
  片側が定数の reif の reified bool を決定変数として探索選択対象に昇格する実験フラグ。
  - パース時(`constraint_registry.cpp`)に `int_le_reif`/`int_lt_reif` で「ちょうど片側が定数」の
    候補を `Model::add_reif_const_bool_candidate(b_id, var_id, threshold)` に記録。
  - presolve(one-hot 集約後)で `Solver::promote_const_reif_bools()` を実行。非定数側変数の
    presolve 後ドメインに対し threshold の相対位置 frac が中央帯 [lo,hi] のときだけ b を決定変数化
    (`unset_defined_var`)、帯外は従属化(`set_defined_var`)。one-hot チャネル bool は対象外。
  - `-R` on のときだけ presolve 後に `build_order` を再実行(off は既存挙動・ゴールデン不変)。
  - 帯は `set_reif_band(lo,hi)` / 環境変数 `SABORI_REIF_BAND_LO/HI` で可変(スイープ用)。
- **A/B ベンチハーネス新設** `benchmarks/minizinc_challenge/bench_promote_reif.py`
  (one_hot ハーネス流用)。`-R -v` の `candidates=N` で対象検出 → baseline vs 各バンドを比較。
  バンドスイープ対応(`--bands`)。健全性(UNSOUND=片方UNSAT/片方SOL)チェック内蔵。
- 全 248 テスト + ゴールデンマスター green。健全性矛盾 0。

### 設計判断・理由

- **eq/ne_reif は昇格対象から除外**: 一律昇格版(初期実装)は 2023-2025 で 13/39 悪化・
  completed x0.829 とネット負け。等値 reif は分岐の情報量が低く決定変数を希釈するだけと判断。
- **le/lt_reif のみ中央帯フィルタ**: threshold が非定数側ドメインの端付近だと b がほぼ即決まり
  情報量ゼロ。中央付近のみ良い二分分岐点になるという仮説。
- **baseline 非対称性に対応**: `int_le_reif` の const-side b は元から defined にされない(decision)
  ため、-R では中央帯外を逆に従属化(demote)する。lt は元から defined なので中央帯のみ昇格(promote)。
- **デフォルト帯 0.35-0.65**: バンドスイープ(0.4-0.6 / 0.35-0.65 / 0.3-0.7 / 0.25-0.75 / 0.2-0.8)で
  completed timing が最良(x1.562, better最多5)。広げると希釈、狭めると勝ち取り逃し。
