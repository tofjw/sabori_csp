## 2026-06-22 (追記) NoGood 伝播のオーバーヘッド削減を Codex 相談 → 全て不採用

**実施内容**
- -C の ng_check/ng_domain 爆発（伝播ストーム）の定数倍削減を Codex に相談
- Codex 提案を実装・計測:
  - ① watch リスト move-out compaction（毎回コピー回避）
  - ② 伝播中 unregister の O(n) erase 回避
  - ③ Literal::is_satisfied の inline 化
  - ④ blocking literal（Codex 自身が「NoGood は極性が逆で意味論が変わる」と除外）

**計測結果（全て不採用）**
- ① move-out compaction: amaze で **約5-10%遅くなった**。bucket が小さい場合、
  per-element の is_watching stale 防御＋write index＋resize＋再lookup の簿記コストが
  「節約できるコピー」を上回る。なお Eq 経路は順序保存（252緑）で健全性は問題なし
- ③ is_satisfied inline: throughput 不変（コンパイラが既に inline 済み）
- 結論: overhead は「短く高活性な Eq conflict 節＝枝刈り本体」に内在し、定数倍最適化でも
  分離不能。全て HEAD(クリーン v1)へ revert

**重大な副次発見（要注意）**
- 作業ツリーに未コミットの紛れ込み編集 `nogood_inactive_restart_limit_ = 50 → 10`
  （solver.hpp）が存在。-C と無関係に NoGood GC を過激化し**ゴールデンを破壊**していた。
  しかも**増分ビルドの stale で間欠的に顕在化**し、「Eq compaction が順序を変える」という
  誤診断で長時間ハマった。クリーン再ビルド(--clean-first)で HEAD=PASS/作業ツリー=FAIL が
  再現し、git diff で 50→10 を特定 → 50 に戻して解決
- 教訓: 挙動差の切り分けは必ず `--clean-first` で。増分ビルドの stale を疑う

**最終状態**
- 作業ツリー = HEAD(3aff86e クリーン v1 -C) と完全一致、252/252 緑
- v1.1 の cap/dedup env ノブも含め overhead 系の試みは全て revert（不採用）

## 2026-06-23 bisect-only -C（mode 出し分け）→ 全15年 net 負けで不採用

**仮説**: bisect 領域（広域変数）は従来 decision-path NG が長大で死に節 → conflict 学習の
価値大。enumerate 領域は従来学習で十分 → -C は overhead のみ。bisect フレームでのみ
conflict 学習する出し分けで net 改善するのでは（env SABORI_CONFLICT_BISECT_ONLY）。

**実施内容**
- bisect フレームのみ conflict 学習する env 出し分けを実装（既定オフ、ゴールデン不変）
- bench_conflict_gate.py で base vs bisect-only -C を minizinc 経由比較

**結果**
- 2022-2025（80問）: wins=10 losses=7 ties=63 → **net +3**（both-modes の net 0 より改善、
  回帰 16→7 に半減）。ただし gfd-schedule(528→245) 等の enumerate 由来の大勝は喪失、
  cgt(51→347→大勝) 等の overhead 由来の負けは回収、というトレードオフ
- **全15年（290問）: wins=20 losses=29 ties=241 → net -9**。複数年が net 負け
- 健全性は一貫 PASS・矛盾ゼロ

**設計判断・理由**
- 2022-2025 の +3 は有利な切り出し（サンプリングの偶然）。大標本では net 負け
- 「狭サンプルの timing/obj@timeout 比較は少数の反転に騙される」を再確認（メモリ既知）
- bisect-gating の着眼は理屈は通るが実効性なし → **不採用、env ゲートも revert**

**結論（-C 系の総括）**
- -C の全バリアント（both-modes=4年 net0 / bisect-only=15年 net-9 / v1.1 cap-dedup=無効 /
  Codex 定数倍最適化=無効〜微減）いずれも field win に届かず。過去 -L/-A/-R と同じ結末
- -C は健全な実験フラグとして既定オフ温存。コードはクリーン v1（both-modes）のまま
- 真の改善は v2（reason tracking→最小決定 NG 上方向伝播+backjumping / LBD / 適応 on-off）が必要
