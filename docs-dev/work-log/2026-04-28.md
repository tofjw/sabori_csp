    - 副作用として gfd-schedule の絶対パス SEGV も同時回避

#### 3. IntOneHotChannel 制約の新規追加

- `mznc2012/solbat/sb.fzn` で `int_eq_reif(x, 0..N-1, b_i)` の集合が同一 x に張り付く one-hot エンコーディングを観測
  - 例: `X_INTRODUCED_77_..97_` 各々に 7 個の int_eq_reif
- MiniZinc 標準にこの構造の global は存在しない（`int_set_channel` は set 向け、`channel(...)` は globalizer 内部用）ことを確認
- **ソルバ独自の集約制約 `IntOneHotChannelConstraint` を追加**:
  - 宣言: `include/sabori_csp/constraints/global.hpp`
  - 実装: `src/core/constraints/global/int_one_hot_channel.cpp`
  - 形式: `bools[i] ⇔ (x == values[i])`、values は実行時定数
  - 集約 pass: `OneHotChannelAggregator`（`src/core/one_hot_channel_aggregator.hpp/cpp`）
    - `Solver::presolve` の Phase 1 後で発火
    - 同一 x に紐付く `IntEqReifConstraint(x, const, b)` 群を発見し、最初の slot を `replace_constraint`、残りを `remove_constraint` + `compact_constraints`
    - その後 `model.build_constraint_watch_list()` で var↔制約マップを再構築（これを忘れて初回 SEGV）
  - propagation: 元の N 個の `IntEqReifConstraint` と等価な伝播を 1 つにまとめ
  - 元の reif 群のラベル（`int_eq_reif:LXXXX`）をカンマ連結して新制約に引き継ぎ、追跡可能に
- **find_value_index の最適化**:
  - aggregator がソート済み values を渡す前提を活かし、`offset_` + `contiguous_` フラグで `v - offset_` の **O(1) 直接添字化**（連続値）/ `std::lower_bound` の **O(log N)** （非連続）に分岐
- **`exhaustive_` (bool) を `holes_` (size_t) に置換**:
  - x の初期ドメインに含まれるが values_ にない値の個数 = "穴"
  - 0 のとき exactly-one、>0 のとき at-most-one として動く
  - 伝播力の弱まりが具体的な数として可視化される
  - gfd-schedule 2018 で 213 制約のうち 171 が `holes_=0`、42 が `holes_=1` の分布を確認
- **IntEqReifConstraint::bump_activity の追加**:
  - y が定数のとき非定数側に full bump（reif 構造を活かす）
- **init_activity / bump_activity の `holes_` 別配分**:
  - exhaustive: x に full bump / `1/var_size` の初期 activity
  - partial: density = (未確定 b 数) / var_size で重み付け（穴に動く分は無駄なので bump を薄める）

#### 4. bump_activity チューニング比較（oh3〜oh7）

| 版 | obj geomean (vs oh2) | time geomean (vs oh2) | 評価 |
|---|---|---|---|
| oh3 | 0.920 | 0.951 | 時間 -5% / obj -8% トレードオフ |
| oh4 | 0.983 | 1.057 | どちらも改善せず（solbat sb_12 で 15× 致命的劣化） |
| oh5 | 0.970 | 1.032 | partial 分岐に整数除算バグで戦略が死亡 |
| oh6 | 0.991 | 1.009 | バグ修正、ほぼ互角 |
| **oh7** | **1.038** | **0.990** | **obj +4% / time -1%、両軸改善** |

- **oh5 で発見した整数除算バグ**: `(var_size - holes_) / var_size` の右辺が両方整数で結果が常に 0 か 1。`static_cast<double>(var_size - holes_) / var_size` で修正
- **oh7 の代表的な改善**:
  - 2022 arithmetic-target: obj 7748 → 1948（4× 改善）
  - 2021 connect: obj 30.2M → 9.6M（3.2×）
  - 2016 solbat sb_13: 19.2s → 4.6s（4.2×）
  - 2025 hitori h5-1: 20.8s → 10.7s（2×）
  - oh6 で致命的だった 2025 stripboard 4.7× 遅延が逆に 3.7× 高速化に転化
- **残課題**: 2012 solbat sb_12 / 2020 minimal-decision-sets / 2019 median-string / 2020 sdn-chain は依然 oh2 が良い

### 設計判断・理由

- **集約は core レイヤで行う**: sabori を汎用 CSP ライブラリとして使うため、aggregator を Python バインディングや直接 C++ API ユーザにも届くよう core presolve に置いた。fzn_sabori 側だけの集約は採用せず
- **検出のしきい値は最低 2 個から**: 同一 x に対する `int_eq_reif` が 2 個以上で発火。partial coverage（exhaustive でない）でも集約する。propagation の正確性は at-most-one で保証
- **values のソートと addr O(1) 化**: aggregator が必ずソート済みを渡す前提でも、直接 API ユーザのために constructor で defensive sort
- **`holes_` (個数) を bool より優先**: 「具体的に何個の穴がある」が伝播力の弱さの量的指標として使えるため、bump 配分のロジックに活きる
- **bump_activity の partial では x の bump を density で薄める**: x が values 外を動く割合だけ b 群への伝播がない、という直感を素直にコード化
