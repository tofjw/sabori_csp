## 2026-02-10 作業ログ

**実施内容**
- `array_bool_and` / `array_bool_or` 制約の伝播バグ修正（前セッションから継続）
  - `r=1` ケースで `return true` が欠落し `r=0` コードにフォールスルーするバグ修正
- hitori が UNSAT になる原因調査
  - `int_lin_eq_reif` の verify エラーを発見（X_800_ の instantiate が制約に通知されない）
  - 根本原因: `ArrayVarIntElementConstraint::propagate_bounds` が `model.remove_value()` を直接呼び出し、イベントキューをバイパスしていた
  - 他の制約（`int_lin_eq_reif` 等）がドメイン変更の通知を受け取れず、不整合な解を「正解」として返していた
- `ArrayVarIntElementConstraint` の修正
  - `propagate_via_queue()` メソッドを新設。`model.enqueue_*` 経由で伝播し、全制約に通知が届くようにした
  - `on_instantiate` / `on_set_min` / `on_set_max` を `propagate_via_queue` を使うよう変更
  - `on_set_min` / `on_set_max` の潜在バグ修正: `vars_[var_idx]`（モデルレベルの変数IDで制約内部配列にアクセス）→ `model.variable(var_idx)` に修正
- デバッグ出力の全削除（`int_lin_eq_reif.cpp`, `solver.cpp`, `solver.hpp`, `main.cpp`）

**設計判断・理由**
- このソルバーはキューベースの伝播アーキテクチャを採用しており、制約は `model.enqueue_*` を使って他変数のドメインを変更すべき。`propagate_bounds` だけが `model.remove_value` を直接呼んでおり、アーキテクチャ違反だった
- `propagate_via_queue` では bounds 伝播のみ行い、個別値の除去は `enqueue_remove_value` / `enqueue_set_min` / `enqueue_set_max` に委譲。キューが後続の伝播を処理するため正しさは保証される
- presolve 時は直接ドメイン操作（`propagate_bounds`）を継続使用（キューがまだ稼働していないため）
