## 2026-06-10 13:00 〜 14:00 circuit 制約の効率化

**実施内容**
- `CircuitConstraint` の伝播強化と効率化（bench_circuit.py で評価）
  - サブサーキット枝刈りを追加: パス結合のたびに「tail → head で戻る値」を `enqueue_remove_value` で除去。presolve でも同様の除去 + 自己ループ除去（n>1 で x[i] ≠ i）
  - AllDifferent の forward checking を追加: x[i]=j 確定時に他の未確定変数のドメインから j を除去
  - `make_circuit`（constraint_registry.cpp）で circuit 変数に `AllDifferentConstraint` を併設（`-G` 時は `AllDifferentGACConstraint`）。Hall ペア / GAC の値伝播が circuit 変数に効くようになった
  - パス管理を union-find（`find()` がパス長に比例）から端点 partner ポインタ方式に変更し全操作 O(1) 化。未確定変数は必ず tail、入次数0ノードは必ず head という不変条件を利用
  - 値プールの `pool_idx_` を `unordered_map` → フラット配列化
  - `bump_activity` を O(n) スキャン → `occupier_` 配列（値 j を占有する変数）で O(1) 特定に変更
  - `on_final_instantiate` を内部状態に依存せずモデル値から閉路を直接たどる検証に変更（解検証パスからも呼ばれるため）
- テスト2件の引数バグ修正: `on_instantiate` の `internal_var_idx` に誤った値（x1 なのに 0）が渡されていた。旧実装の `find()` が偶然吸収していたが、新実装の O(1) 化は「i は自パスの tail」という不変条件に依存するため顕在化

**設計判断・理由**
- 勝敗を分けたのは伝播強化（枝刈り + FC + AllDifferent 併設）。circuit にはこれまで値レベルの伝播が一切なく、閉路を作ってから失敗検出するだけだった。「未解→解」「TIMEOUT→OPTIMAL」レベルの変化はこの3つによる
- AllDifferent は circuit 内に Hall 伝播を再実装せず、既存のテスト済み制約を registry で併設する形を選択（コード重複回避、GAC フラグも自動で効く）
- circuit 自身の FC は AllDifferent 併設後は一部冗長だが、C++ API / Python バインディング経由で circuit 単体使用されるケースのために残した（重複 enqueue は contains チェックで実質 no-op）
- bump_activity の新旧は bump 対象集合が同一で jitter(rng) の消費順だけが違う。マージナルな問題（cvrp / yumi-static 等）はこのノイズレベルで勝敗が反転するため、構成選択は単発ベンチではなく「未解→解」等の頑健な変化で判断した
- partner 方式の正しさ: 未確定変数 i は出エッジがないので必ず自パスの tail、入次数0の j は必ず head。よって `partner_[i]` / `partner_[j]` だけで結合・閉路判定が O(1) で完結し、trail も旧 partner 値の復元のみで巻き戻せる
