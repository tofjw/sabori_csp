  - **根本原因**: reachability BFS のエッジ方向が逆（G_M 方向で BFS していたが、Reverse(G_M) 方向が正しい）

**設計判断・理由**

- GAC presolve 削除: presolve 中は `model.is_instantiated()` 等の SoA API が未更新で、`find_maximum_matching` が不正な結果を返す。presolve/propagation の API 境界を跨ぐ呼び出しは危険
- BFS 方向修正: Régin のアルゴリズムでは free value から Reverse(G_M) 方向の到達可能性が必要。従来実装は G_M 方向で BFS しており、free value の出辺がゼロのため何も到達不能と判定。SCC が部分的に補正していたが特定構造で false pruning 発生

- Tarjan SCC 内でも隣接リストを毎回構築している（O(n*m)）。`compute_sccs_and_filter` 冒頭で1回構築して共有すべき

- `compute_sccs_and_filter` のパフォーマンス改善（前回の BFS 方向修正で発生した O(n*m) ホットスポットの解消）
  - `val_to_vars_` 逆引きインデックスを `compute_sccs_and_filter` 冒頭で1回構築

  - `tarjan_dfs` シグネチャを `(Model&, int)` → `(int, const vector<vector<int>>&)` に変更
  - `val_to_vars_`, `adj_` をメンバ変数化して毎回のヒープ確保を回避
- sudoku_fixed ベンチマーク: 120s → 49.6s に改善（目標 60s 以内達成）
- AllDifferentGAC の正確性テストを15件追加（全解列挙+ブルートフォース照合）

**設計判断・理由**
- 逆引きインデックス `val_to_vars_` を BFS と Tarjan 隣接リスト構築の両方で共有し、ドメイン走査を1回に集約

- テストは全解列挙 + ブルートフォース期待値で false UNSAT / over-pruning の両方を検出する設計

- fzn_sabori のデフォルトを NoGAC に変更、`-G` オプションで GAC を選択可能に

- NoGAC をデフォルトにした理由: ベンチマーク全体では NoGAC の方が安定して速い。GAC は特定の問題（sudoku のような離散ドメイン SAT 問題）でのみ有利
