## 2026-06-27 activity bump 源の分解計測（冗長な代替の発見）と §3/§4 訂正

**実施内容**
- 前日の §3「NoGood は activity 経由で効く（+18）」の内訳を、コードのスケール差をきっかけに深掘り。
  - 発端: NoGood 由来の activity bump は2系統——学習時（`nogood_manager.cpp:223`, `activity_inc × 0.01` の極小スケール）と伝播時（`:74/100/122`, `activity_inc/n` フルスケール）。制約 bump（jitter 入りフルスケール）・決定変数 bump（`handle_failure`, フルスケール常時ON）も別系統。

  - `SABORI_NG_LEARN_BUMP=0` / `SABORI_NG_PROP_BUMP=0`: NoGood の学習時/伝播時 bump を別々に off（`NoGoodManager::set_learn_bump`/`set_prop_bump`、`activity_bump_enabled_` を learn/prop に分割）

- bench 2本: `bench_ng_bumpsplit.py`（単体3つ）、`bench_ng_compensation.py`（単体 + all3_off + everything_off の6構成）。38問×5シード・目的値判定。
- 結果（compensation bench, 各 full に対する悪化 net）:
  - learn_off +7 / prop_off +2 / decvar_off +8（単体はどれも小）
  - all3_off +14 / everything_off（制約bumpも） **+21（全シード正）**

- §3 に「冗長な代替」小節を新設、§4 の base/none を「generic すら none を超えず＝機構ごと余剰」に訂正（JP/EN 全文版）。まとめ表・結論バレットも連動修正。

**設計判断・理由**
- **全員の予想が外れた**: ①ユーザー仮説「0.01 学習 bump は jitter に消える」→ 単体 +7 で**消えない**（最小スケールが伝播 bump より効く）。②私の仮説「伝播 bump が +18 の主役」→ 単体 +2 で**最小**（主役でない）。③「決定変数 bump が最大源」→ 単体 +8 止まり。**per-event スケールは効き目を予測しない（累積 = 大きさ×頻度×範囲。学習 bump は毎 conflict・全 trail 変数に効く）**。
- **冗長な代替モデル**: activity 供給は複数経路が互いに代替可能。1つ抜いても残りが補償（単体≈小）、全部抜くと補償しきれず +21。「どのチャネルが効くか」でなく「ある程度の量があるか」。big thesis（activity による傾向制御は効く）はむしろ**補強**（全停止 +21・全シード正）。
- **§4 との一貫**: 制約 bump も代替の一員。他があれば余剰（§4 で none≳base）、全部切った後は +7 効く（everything_off vs all3_off）。「他があれば冗長、なければ効く」の補償。「base は妥当なベースライン」は撤回。既定 structural は3つ中最弱で見直し余地。
- **目的値判定の重要性（再確認）**: everything_off は `fails` で激減（一見「速い」）だが目的値では最悪（+21）。productive に探索せず失敗だけ減っただけ。**node 数で測ると逆結論**を出すところだった。
- **run 間ブレ**: `ng_full vs ng_prune=+18`（前日 run）と「all3_off=+14」が一致しないのは 30s wall-clock 判定の分散。**同一 run 内の単調性と全停止 +21・全シード正が頑健な事実**で、個々の net 絶対水準は run で揺れる。

## 2026-06-27 temporal による activity マスクの発見と「activity 主役」へのフレーミング修正

**実施内容**
- 指摘①: temporal_activity はカウンタだが、選択時には未割当 hot が通常 0〜1 個で**古典 Last Conflict のフラグ的に振る舞う**。一時計装で確認（未割当限定 multi_hot: test-scheduling 1.2% / code-generator 4.7% / gametes 12% / elitserien 40%。値は最大 737 まで累積するが選択には効かないことが多い）。problem-dependent。計装は計測後に撤去。
- 指摘②（核心）: temporal はバックトラック直後の最初の1手を決めるだけで、**降下の大半は activity が駆動**。なのに「activity がほとんど効いていない/冗長」と読める記事の書き方は誤り。
- 検証: `bench_temporal_mask.py`（4構成・同一 run）。activity bump 全停止の悪化を temporal ON/OFF で比較:
  - **full vs full_actoff（temporal ON）= +18**
  - **t_off vs t_off_actoff（temporal OFF）= +81**（全シード +14〜+19、full が 94–13）
  - → **activity の真の寄与は +81。temporal（とフォールバック MRV）が大半をマスクし、フル構成の marginal ablation が +18/+21 に圧縮されていた**。
- 記事修正（JP/EN）: §1 に「temporal=再開地点の override、activity=降下の駆動・主役、+81 にマスク」を追加。§3 に「+21 は marginal・masked、真値 +81」の但し書き。capstone/結論/まとめ表/TL;DR を「2軸が探索フェーズで分担（temporal override / activity 降下駆動）」に修正。activity を「タイブレーク」と書いた箇所を「降下の主役」に格上げ。

**設計判断・理由**
- **強い第1基準が下層の評価をマスクする**: §3 の「補償」の一段大きい版。ablation は「他の機構が ON のときの marginal 値」を測るので、上位の強い機構（temporal）が下位（activity）を覆うと過小評価される。**マスクし合う系では、対照を変えて両方向から測らないと主役を見誤る**（今回 temporal OFF を baseline にして発覚）。
- **「最強の単一レバー=temporal(+25)」は marginal の話**で、activity の unmasked 寄与(+81)の方が大きい。記事は両方を併記し、temporal=override / activity=降下主役、と labor 分担で書く。
- 「冗長な代替」は activity を**供給する bump 源**の代替性に限定。activity 選択そのものは主役、と明確化。
