**設計判断・理由**
- **全員の予想が外れた**: ①ユーザー仮説「0.01 学習 bump は jitter に消える」→ 単体 +7 で**消えない**（最小スケールが伝播 bump より効く）。②私の仮説「伝播 bump が +18 の主役」→ 単体 +2 で**最小**（主役でない）。③「決定変数 bump が最大源」→ 単体 +8 止まり。**per-event スケールは効き目を予測しない（累積 = 大きさ×頻度×範囲。学習 bump は毎 conflict・全 trail 変数に効く）**。
- **冗長な代替モデル**: activity 供給は複数経路が互いに代替可能。1つ抜いても残りが補償（単体≈小）、全部抜くと補償しきれず +21。「どのチャネルが効くか」でなく「ある程度の量があるか」。big thesis（activity による傾向制御は効く）はむしろ**補強**（全停止 +21・全シード正）。
- **§4 との一貫**: 制約 bump も代替の一員。他があれば余剰（§4 で none≳base）、全部切った後は +7 効く（everything_off vs all3_off）。「他があれば冗長、なければ効く」の補償。「base は妥当なベースライン」は撤回。既定 structural は3つ中最弱で見直し余地。
- **目的値判定の重要性（再確認）**: everything_off は `fails` で激減（一見「速い」）だが目的値では最悪（+21）。productive に探索せず失敗だけ減っただけ。**node 数で測ると逆結論**を出すところだった。
- **run 間ブレ**: `ng_full vs ng_prune=+18`（前日 run）と「all3_off=+14」が一致しないのは 30s wall-clock 判定の分散。**同一 run 内の単調性と全停止 +21・全シード正が頑健な事実**で、個々の net 絶対水準は run で揺れる。
