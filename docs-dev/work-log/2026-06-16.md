  find(v)==end() パターン6箇所を private inline is_removed() に一本化。

**設計判断・理由**
- removed_set_ の unordered_set→ビットマップ化は見送り(確定): bounds-only はレンジ>10000
  専用で上限ガード無し、巨大レンジ(例 0..2^31)でビットマップ即確保はメモリ激増(〜268MB/変数)。
  遅延確保の set の方が最悪ケース安全。自動選択はフラグ分岐増で衛生目的に逆行、profile 上 ROI も低い。
- BoolDomain / variant 化 / remove_value O(n) / rewind virtual は純リファクタでなく性能最適化。
  profile-20260614 の結論(性能 ROI は propagator 本体[IntLinEq 差分/TTEF]と核ループ定数倍
  [process_queue/set_min-max])より別イニシアチブへ分離。
