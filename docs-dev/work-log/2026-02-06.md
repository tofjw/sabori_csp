## 2026-02-06 Solver リファクタリング + TSP 無限ループバグ修正

**実施内容**
- `Solver::propagate` → `propagate_instantiate` にリネーム
- `process_queue` リファクタ: `need_propagate` フラグを廃止し、各 case 内で直接 `propagate_instantiate` を呼ぶように変更
- `Solver::propagation_queue_` を廃止し、Model の `pending_updates_` に一本化
  - `pending_updates_` を `vector` → `deque` に変更
  - `pending_updates()` を `has_pending_updates()` + `pop_pending_update()` に置き換え
  - `Solver::enqueue_instantiate` を削除、NoGood 伝播から `model.enqueue_instantiate()` を直接呼び出し
- solver.cpp 内の `var->min()`/`var->max()`/`var->is_assigned()` 等を `model.var_min()`/`model.var_max()`/`model.is_instantiated()` に置き換え（SoA 直接アクセスで高速化）
- `enqueue_*` の O(n) 重複チェックスキャンを削除（`process_queue` 側の既存ガードで重複を O(1) スキップ）
- **TSP 無限ループバグ修正**: `int_lin_eq::propagate()` 内で `var->remove()` でドメインを直接変更した後に `model.var_min()`/`model.var_max()`（SoA）を参照していたため、SoA が更新されず差分更新が常に 0 → `while (changed)` 無限ループ。`var->min()`/`var->max()` に戻して修正

**設計判断・理由**
- `propagation_queue_` と `pending_updates_` の二重キュー構造は不要だった。制約から model に enqueue → solver が model から pop する一本化で十分
- `enqueue_*` の重複チェックは enqueue ごとに O(n) のコスト。`process_queue` 側に `new_min <= prev_min` 等のガードが既にあるため、重複は O(1) でスキップされる
- `propagate()` 内でドメインを直接操作する場合、SoA は更新されないため `model.var_min()` ではなく `var->min()` を使う必要がある

**気づいた課題・TODO**

- `propagate()` 内のドメイン直接操作と SoA の不整合問題は、将来的に `propagate()` を `enqueue_*` 方式に完全移行することで根本解決すべき

## 2026-02-06 reif 制約 presolve バグ修正

**実施内容**
- cgt ベンチマークが誤って UNSAT を返すリグレッションを調査・修正
- `IntLinEqReifConstraint::presolve()`, `IntLinNeReifConstraint::presolve()`, `IntLinLeReifConstraint::presolve()` がキャッシュ値（`current_fixed_sum_`, `min_rem_potential_`, `max_rem_potential_`）を参照していたのが原因
- Phase 1 はイベント処理が組み上がる前なのでキャッシュは未初期化（すべて0）。`min_sum = max_sum = 0` となり、`target != 0` の制約で不正に b を推論 → false UNSAT
- 3つの reif 制約の `presolve()` を、変数ドメインから毎回 min_sum/max_sum を計算する方式に修正
- `Solver::presolve()` の Phase 1 前に `model.prepare_propagation()` を追加する案は採用せず（根本原因はキャッシュ依存）

**設計判断・理由**
- `presolve()` (旧 `initial_propagate`) は固定点ループ内で繰り返し呼ばれる。他の制約がドメインを変更するため、キャッシュ値は常に古くなる可能性がある
- `prepare_propagation()` で初期化されるキャッシュ値は、イベント駆動の `on_instantiate` で差分更新される前提の設計。`presolve()` ではこの前提が成り立たないので、毎回ドメインから計算するのが正しい
- `IntLinEqConstraint::presolve()` は元々毎回計算していたので、reif 版もそれに揃えた形

## 2026-02-07 02:48 パフォーマンスホットスポット対策 実装

**実施内容**
- プロファイリングで特定された7つのホットスポットに対策を実装

  2. 伝播キューを `std::deque` → `std::vector` + read index に変更

**設計判断・理由**

- `sparse_ref()` は全箇所で `index_of()` に置き換え、メソッド自体を削除
