## 2026-06-15 リファクタリング Phase 3: 機械的重複削減（SparseSetPool / ConstraintTrail）

**実施内容**

- **SparseSetPool クラス抽出**（commit `5256c19`）
  - `include/sabori_csp/sparse_set_pool.hpp` 新規。可逆 Sparse Set を共通化。
  - all_different / all_different_except_0 が個別に持っていた `pool_values_` / `pool_sparse_`
    (unordered_map) / `pool_n_` のスワップ削除ボイラープレートを `pool_.remove()` /
    `contains()` / `consumed()` / `value_at()` に置換（constraint 層で net ~115 行削減）。
  - sparse 写像は値域が密なら offset 付きフラット配列、疎で広域なら unordered_map を
    自動選択（計画の「unordered_map フラット配列化」を安全に実現）。
  - AllDifferentGAC サブクラスの直接メンバ参照（`is_val_in_pool`, GAC スナップショット
    構築）を `pool_` アクセサに追従。
  - `tests/cpp/test_sparse_set_pool.cpp` 新規: 参照モデル（std::set）とのランダム照合。
    flat/map 両パス、restore 可逆性を 33,518 assertion で検証。
- **ConstraintTrail<State> テンプレート抽出**（commit `b803462`, `65eb553`）
  - `include/sabori_csp/constraint_trail.hpp` 新規。「同一セーブポイントでは一度だけ保存」
    「save_point 超えを巻き戻す」trail 定型を `save_if_needed()` / `rewind_to(restore_fn)` /
    `clear()` に共通化。
  - int_lin_eq/le/ne + reif/imp の計7制約（`b803462`）、all_different 系2制約（`65eb553`）に適用。
    push/pop/back の手続きを排除し、復元は各制約のラムダで明示。
- **検証**
  - 全コミットで ctest 205/205 green（golden master = 182 fzn の `-a -s` 出力 byte 一致）。
  - SparseSetPool は alldiff bench ゲートも実行: Sabori Wins 6 / Timeouts 0 で
    baseline-20260614 と完全一致（退行なし）。

**設計判断・理由**

- unordered_map フラット化は memory 上「性能 ROI 低（flat profile に出ない）」のため、
  主目的は**重複削減（設計衛生）**と位置づけ。フラット配列は offset ベースだが、値域が
  広すぎる場合（span > max(1024, 4×|values|)）は memory blowup を避けて unordered_map へ
  フォールバックする自動選択にした。
- ConstraintTrail は hot path（IntLinEq の bound 差分計算）に**不介入**。trail は
  イベント/バックトラック毎の O(1) 簿記のみで、復元ラムダは Release で inline 展開される
  ため定数倍劣化なし。純リファクタなので golden master を主検証とした。
- **circuit の SparseSetPool 化は見送り**。circuit は元から内部インデックス 0..n-1 用の
  フラット配列（unordered_map 不使用）で、trail が union-find 状態と密に絡む。dedup 利得が
  小さく false UNSAT リスクが高いため、Phase 3 のスコープから外した。

- **死コード処分**（commit `9ccab04`）: 呼び出し元のない private メソッド3件を削除
  （AllDifferent/Except0 の `remove_from_pool`、IntLinEq の `check_feasibility`）。
  grep で src/include/tests/python 全体に caller がないことを確認。
  - bump_activity 系の `#if 0` ブロック群（constraint.cpp / increasing.cpp / comparison.hpp）は
    activity ヒューリスティクスの A/B 実験アーカイブ（日付ラベル付き・active なチューニング
    領域）であり停滞死コードではないため**保持**。AllDifferentGAC も保持。
  - → これで Phase 3（機械的重複削減・死コード削除）は一通り完了。

## 2026-06-15 (続き) Phase 4 着手: LinearConstraintBase

**実施内容**

- **LinearConstraintBase 新設**（commit `0bde8a4`）
  - int_lin_* 7制約（eq/le/ne + reif/imp）のコンストラクタが個別複製していた線形項集約
    （同一変数の係数合算・係数0除外・unique_vars 構築）を `aggregate_terms` に共通化。
    係数列 `coeffs_` も基底へ移動。全7制約を `: public LinearConstraintBase` に変更。
  - 非 reif(eq/le/ne) の冗長な inner-loop（VariablePtr=Variable* なので shared_ptr 探索は不要）を
    排除。reif/imp の b 追加ロジックも aggregate_terms 後の2行に集約。
  - ~80行削減。

**設計判断・理由**

- **potential 管理(min_rem/max_rem/fixed_sum)の差分更新は基底化しない判断**:
  potential の形状が eq=min+max / le=min のみ / ne=fixed_sum+unfixed_count と分岐し、単一基底に
  きれいに収まらない。かつ on_instantiate/on_set_min/on_set_max という **hot path**（IntLinEq は
  celar で 60%）に介入するため、限界的な dedup のために定数倍リスクを取らない。集約（constructor、
  1回限り）のみで安全に ROI を確保した。
- 検証は golden master を主とした。var_ids_ の順序（unordered_map 反復順）が完全保存された
  ことを 205/205 green が保証（順序が変われば golden は全面赤になる）。constructor のみの変更で
  hot path 不介入。

## 2026-06-15 (続き) Phase 4: ReifConstraintBase — eq_reif/ne_reif 統一

**実施内容**

- **ブルートフォース安全網を先行構築**（commit `6957db3`, `tests/cpp/test_lin_reif_brute.cpp`）
  - int_lin_eq_reif / int_lin_ne_reif のランダム小規模インスタンスを `solve_all` で全解列挙し、
    総当たり参照列挙と**解集合一致**を照合。b-free / b-fixed(0/1) / 負係数 / 重複変数（aggregate
    経路）を網羅。806 assertion が**リファクタ前の現行コードで green**であることを確認してから着手。
  - 計画の鉄則「reif 変更には false UNSAT を捕捉するランダム全解照合を併設」を満たす。
- **IntLinEqNeReifBase で eq_reif/ne_reif を統一**（commit `12aad80`）
  - 両者は内部状態（fixed_sum / min,max ポテンシャル / unfixed_count / trail）と差分更新が
    **byte 一致**で、述語 P=(sum==target) の極性のみが異なる。`negated_` フラグで出し分ける基底に集約。
  - b<->述語 のロジック（b 未確定→推論 enqueue、b 確定→矛盾検出）をファイル内で約5回複製していたのを
    `reconcile_b(min_sum,max_sum)` に一本化。`b_true_means_P = !negated_` で eq/ne を代数的に統一。
  - 公開2クラスは name() + コンストラクタフラグだけの薄い派生に。registry / model_simplifier の
    dynamic_cast / pybind11 バインディングは無変更（型は別個のまま保持）。
  - net **~470行削減**（source 748→335, header -58）。

**設計判断・理由**

- **挙動差は「観測上 no-op」と確認した上で安全側に統一**:
  - ne_reif は元々 `Constraint::on_instantiate`（2WL 簿記）を呼んでいなかった（eq/le は呼ぶ）。
    eq/ne_reif は on_last_uninstantiated を持たず watch を伝播に使わないため出力不変 → 呼ぶ側に統一。
  - on_final のガード・確定後の冗長再チェックは eq 版（厳密に安全側）を採用。到達時は必ず true を返す
    no-op であることを代数的に確認。
  - prepare_propagation の初期化を `model.is_instantiated/value`（propagation callback の推奨 API、
    protein リグレッションの教訓）に統一（ne は従来 `is_assigned` だった）。初期化時は両者整合。
  - 空係数時の極性差（eq は b を確定させず check のみ / ne は確定させる）は presolve 後到達不能な
    防御経路だが、挙動を変えないため `negated_` で明示的に保存。
- **le_reif/le_imp は対象外**: b 確定時に線形変数を `propagate_bounds_le/gt` で刈り込む形状で、
  eq/ne（線形側は刈らず b 推論のみ）と構造が異なる。le 用の brute net を併設してから別途検討。
- 検証は golden master（182 fzn の `-a -s` byte 一致＝探索軌道不変）を主、brute 806 を副とした。
  golden byte 一致のためベンチゲートは不要（軌道不変の最強証拠）。

## 2026-06-15 (続き2) Phase 4: le_reif/le_imp — 共通刈り込みカーネル抽出

**実施内容**

- **brute net を le 系へ拡張**（commit `8d555b0`）
  - `test_lin_reif_brute.cpp` を `Rel` enum（EqReif/NeReif/LeReif/LeImp）に一般化。le_reif の
    双条件、le_imp の含意（b=1→sum<=bound）を参照列挙で網羅。randomized 4 suite + b 固定の
    刈り込みエッジ。1614 assertion を**現行コードで green** にしてから着手。
- **`LinearConstraintBase::prune_sum_le` 抽出**（commit `fe736eb`）
  - `sum<=bound` の境界絞り loop が IntLinLe::propagate_bounds / IntLinLeReif::propagate_bounds_le /
    IntLinLeImp::propagate_bounds で**3 重に逐語複製**されていたのを基底のカーネルへ一本化。
    3 箇所とも委譲に置換。int_lin_le は entailment 早期 return を残してから loop を委譲。
  - カーネルは先頭 `coeffs_.size()` 個を走査 → reif/imp は末尾 b を除外、素の le は b 無しで
    `var_ids_.size()` と一致するため3者で挙動同一。net **-48行**。

**設計判断・理由**

- **le_reif/le_imp の全クラス統一は見送り**。le_imp は le_reif の「b=1 半分」で、
  状態形状が違う（min ポテンシャルのみ・max_rem 無し）、推論も対偶（min_sum>bound→b=0）の
  一方向のみ。統一すると le_imp に未使用の max_rem 追跡と毎コールバックのフラグ分岐を hot path に
  持ち込み、eq/ne 統一で確認した「hot path に限界的 dedup のため介入しない」原則に反する。
  → 真の重複（刈り込みカーネル＝最大の共有ブロック）だけを抽出する方が ROI が高く安全。
- gt 方向（`propagate_bounds_gt`）は le_reif にしか無く重複が無いため対象外。
- 検証は golden byte 一致を主、brute 1614 を副。golden 一致のためベンチゲート不要。

## 2026-06-15 (続き3) Phase 4: comparison.cpp presolve ヘルパ抽出

**実施内容**

- **scalar comparison brute net 新設**（`tests/cpp/test_cmp_reif_brute.cpp`）
  - int_eq/ne/lt/le + reif/imp を `Cmp` enum で網羅。x,y 範囲を重なり/分離/包含で総当たりし、
    b は free/0/1 全極性。参照列挙と解集合一致を照合。1296 assertion を現行コードで green 確認。
- **presolve ヘルパ2種を抽出**（comparison.cpp）
  - `intersect_eq(model, x, y, changed)`: x==y のドメイン交差（bounds 版 / sparse 版）。
    IntEq::presolve / IntEqReif::presolve(b=1) / IntEqImp::presolve(b=1) の**逐語3複製**を一本化。
  - `enforce_le(model, lo, hi, strict, changed)`: lo<=hi（strict で lo<hi）を直接操作で強制。
    IntLt(strict) / IntLe / IntLeReif(b=1: x<=y) / IntLeReif(b=0: x>y を lo=y,hi=x,strict で表現)。
    le_reif b=0 の「y<x」は enforce_le の引数入替＋strict で元ロジックと一致。
  - net **-69行**（141 del / 72 ins、ヘルパ込み）。

**設計判断・理由**

- ヘルパは comparison.cpp 内の無名 namespace に file-local 配置（これらの制約は Constraint 以外の
  共通基底を持たないため、基底メソッド化せず free function が素直）。直接ドメイン操作（remove_above/
  below, remove）は presolve のみで許される（CLAUDE.md / memory のパターン）ため presolve 専用。
- propagation 側（on_instantiate/on_set_min/max の相互伝播）は b ロジックと絡み presolve ほど
  逐語的でないため今回は対象外。
