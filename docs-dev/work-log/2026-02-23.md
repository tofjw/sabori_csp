## 2026-02-23 is_defined_var 自動検出の拡張

**実施内容**
- atsp.fzn の FlatZinc `is_defined_var` アノテーションを分析し、int型変数（35個）が探索性能に最も critical であることを特定
  - `FZN_DEFVAR` 環境変数による選択的有効化で検証: none→33M fails(TIMEOUT), int→1,257 fails, bool→753K fails, all→662 fails
- critical な変数の定義制約を分類: int_times(10), int_lin_eq(8), int_max(5), array_int_element(5), bool2int(4), array_var_int_element(4)
- 以下の制約で結果変数の `defined_var` 自動検出を追加:
  - `int_times(x,y,z)` → z を defined_var に
  - `int_max(x,y,m)` / `int_min(x,y,m)` → m を defined_var に
  - `int_abs(x,y)` → y を defined_var に
  - `bool2int(b,i)` → i を defined_var に（非エイリアス時）
  - `array_int_element` → 全定数配列で result を defined_var に（単調限定から拡張）
- substitution の X/Y スワップ修正: X 候補が protected（output_var）の場合、Y が |coeff|=1 なら入れ替えて消去を実行

**設計判断・理由**
- 関数的制約（int_times, int_max 等）の結果変数は入力が決まれば一意に定まるため、無条件で defined_var にしてよい（入力が defined_var でも問題ない）
- int_lin_eq ヒューリスティックの `any_defined` チェックは維持（除去すると TSP で decision variable が誤って defined になるリグレッション発生）
- array_int_element は定数配列なので index が決まれば result は常に一意（単調性は不要）
- substitution で X が protected なら Y にスワップすることで、TSP の `idx_i = next_i + offset` の `idx_i` が消去され、10変数分の探索空間削減

- substitution のタイブレーク（両方 is_defined_var かつ同ドメインサイズ）で FZN `defines_var` 制約アノテーションを使えば更に正確な X 選択が可能

## 2026-02-23 alldifferent_except_0 native 制約の有効化実験と撤回

**実施内容**
- `alldifferent_except_0` を MiniZinc から native 制約として呼び出せるよう有効化
  - `redefinitions.mzn` に `fzn_alldifferent_except_0.mzn` を追加
  - `model.cpp` にパーサ登録、`global.hpp` にクラス宣言追加、`CMakeLists.txt` にソース追加
- concert-hall-cap ベンチマークで分解版との性能比較実験:
  - `{a,0}→0` 伝播: x[i]=a 確定時、ドメインが {a,0} の x[j] を 0 に確定 → obj=36214 (14解)
  - full removal: x[i]=a 確定時、全 x[j] から a を除去 → obj=36214 (24解)
  - 分解版 (no-ad0): obj=36482 (fails=665万)
  - 伝播なし (元): obj=35954
- 分解版の伝播は「x[j] が 0 を取れない場合のみ a を除去」で native full removal より弱いが、制約密度（O(n²) ペアワイズ制約）が変数選択ヒューリスティックに有利に作用し、結果的に良い解を発見
- native 版にアドバンテージなしと判断し、`redefinitions.mzn` から除外、forward propagation コードも削除

**設計判断・理由**
- native 制約の伝播は分解版より強い（無条件 removal vs 条件付き removal）にもかかわらず、探索性能で劣る
- 主因は制約密度: 分解版は変数あたり最大23制約（ヒューリスティックの手がかりが多い）vs native は最大7制約
- 現状のヒューリスティック（制約数ベース）では、グローバル制約化のメリットが出ない
