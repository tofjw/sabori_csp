## 2026-02-11 02:06 comparison.cpp on_set_min/on_set_max 実装

**実施内容**
- comparison.hpp の 8 クラス（IntNeConstraint 以外）に `on_set_min` / `on_set_max` の宣言を追加
- comparison.cpp に各制約の bounds 伝播ロジックを実装:
  - **IntLtConstraint** (x < y): x.min↑ → y.min ≥ x.min+1, y.max↓ → x.max ≤ y.max-1
  - **IntLeConstraint** (x ≤ y): x.min↑ → y.min ≥ x.min, y.max↓ → x.max ≤ y.max
  - **IntEqConstraint** (x = y): 双方向の min/max 伝播
  - **IntMaxConstraint** (m=max(x,y)): x/y bounds → m bounds, m.max↓ → x,y max
  - **IntMinConstraint** (m=min(x,y)): m.min↑ → x,y min, x/y bounds → m bounds
  - **IntLeReifConstraint**: bounds で b 確定 + b 確定時の bounds 伝播
  - **IntEqReifConstraint**: disjoint bounds → b=0, b=1 時の eq 伝播
  - **IntNeReifConstraint**: disjoint bounds → b=1, b=0 時の eq 伝播
- IntNeConstraint は対象外（bounds では伝播不可）
- 全 110 テスト pass

**設計判断・理由**
- instantiate 時のみの伝播では bounds 変化が即座に反映されず、探索空間の枝刈りが遅れていた
- bounds-only な制約（Lt, Le, Max, Min）は最も効果が高い
- reif 制約は bounds で b の値を早期決定できるケースがある
- IntNeConstraint は domain consistency の問題のため on_set_min/on_set_max は不要

## 2026-02-11 save_partial_assignment O(1) 高速化

**実施内容**
- `save_partial_assignment` の O(n) → O(1) 高速化
  - `Model` に `instantiated_count_` メンバを追加し、instantiated 変数数をインクリメンタルに管理
  - `instantiate()`, `set_min()`, `set_max()`, `remove_value()`, `rewind_to()`, `add_variable()`, `sync_from_domains()` の計7箇所でカウンタを更新
  - `Solver::save_partial_assignment` のカウントループを `model.instantiated_count()` に置き換え
- 全 126 テスト pass

**設計判断・理由**
- cgt プロファイリングで `save_partial_assignment` が **16.84%** (3.82s, 316万回呼出) を占めていた
- 大半は `best_num_instantiated_` 以下で early return するが、カウントループ自体が O(n) でコスト大
- Model 側でカウンタを管理すれば、ドメイン操作の各箇所に `+1`/`-1` を追加するだけで O(1) 判定が可能
- `rewind_to` では復元前後の instantiated 状態を比較して双方向調整
