**実施内容**
- IntLinLe の `propagate_bounds()` に entailment detection を追加
  - max remaining sum <= slack なら propagation をスキップ（early-break 付き）
  - 98% の無駄な呼出し（reduction なし）を回避
- improvement probe の maximize 時の target 計算を修正
  - 旧: `obj_ub - improvement + 1`（上限基準 → best から遠すぎる target）
  - 新: `obj_lb + improvement`（下限基準 → best 付近から探索）
  - roster-shifts-bool で検証: 修正前は30秒で解1個、修正後は11回改善

**設計判断・理由**
- IntLinLe entailment: propagate_bounds 冒頭で O(n) スキャンだが、entailed でない場合は early-break でほぼゼロコスト。インクリメンタル管理（trail 追加）より変更最小・バグリスク最小
- probe target: maximize で obj_ub 基準だと UNSAT 時のドメイン縮小が 5% しかない。obj_lb 基準なら UNSAT 時に 95% カットでき、SAT 時も best 付近の現実的な改善を発見できる
