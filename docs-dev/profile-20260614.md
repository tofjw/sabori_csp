## 結論（フェーズ5 優先順位への含意）

1. **hot path は問題クラスで支配項が全く異なる**: 線形系=`IntLinEqConstraint` 差分伝播、
   スケジューリング系=`TTEFPropagator` forward/backward。汎用的な単一ボトルネックは無い。
2. **計画/MEMORY が挙げる 2026-02 の Domain 系残件は今回どの上位にも現れない**:
   `BoundsDomain::removed_set_`(unordered_set)、`remove_value` O(n)、
   `rewind_dirty_constraints` virtual dispatch は flat profile 上位 ~17 関数に不在。
   → **フェーズ5 の「Domain 表現刷新」を最優先とする根拠は現データには無い**。
   removed_set_ ビットマップ化等は「設計衛生」としては有効でも性能 ROI は低い見込み。
3. **性能 ROI が高いのは制約 propagator 本体**:
   - `IntLinEqConstraint::propagate_*`（差分伝播のホットさ）→ フェーズ4 LinearConstraintBase の
     リファクタは**この hot path を絶対に劣化させない**こと（ゴールデン + ベンチゲート厳格）。
   - `TTEFPropagator::forward/backward_pass`（scheduling の 9 割）→ 内部ソート/再計算の削減が候補。
   - `Solver::process_queue` は全問題で 6〜8%（伝播ディスパッチの定数倍）。
4. `Model::set_min/set_max` が celar で 13% = 境界更新は inline/分岐削減の価値あり（核ループ）。

**推奨**: フェーズ5 を「Domain 表現」中心から「propagator hot path（IntLinEq 差分 / TTEF）+
process_queue/set_min-max の定数倍」中心へ組み替える。Domain の unordered_set 除去は
衛生目的（フェーズ3/5）に留め、性能根拠とはしない。
